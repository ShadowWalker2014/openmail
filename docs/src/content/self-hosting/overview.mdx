# Self-Hosting Overview

OpenMail is designed to be self-hosted. Every component is open-source and deploys via Docker or standard Node.js processes.

## Required Infrastructure

| Component | Description | Minimum |
|-----------|-------------|---------|
| PostgreSQL | Primary database | 14+ |
| Redis | Queue and cache | 6+ |
| Node.js / Bun | Runtime | Node 22 / Bun 1.x |

## Services

## Deployment Options

**[Railway](/self-hosting/railway)** вҖ” Recommended for production. Deploy each service from the same GitHub repo with automatic builds and managed Postgres + Redis.

**[Docker Compose](/self-hosting/docker)** вҖ” Great for local development, testing, or simple single-server deployments.

**Kubernetes** вҖ” Supported but not officially documented. Each service has a Dockerfile and can be containerized independently.

## Single-Tenant vs Multi-Tenant

OpenMail supports both:

**Single-tenant (most self-hosters):** One installation, one company. All users belong to one or a few workspaces. Use the same Resend API key for everything.

**Multi-tenant (hosted SaaS):** Multiple independent customers. Each workspace gets its own Resend API key and sender domain configured in the dashboard.

## Resource Requirements

For a small team (< 50k contacts, < 100k emails/month):

- API: 256MB RAM, 0.25 vCPU
- Worker: 256MB RAM, 0.25 vCPU  
- Tracker: 128MB RAM, 0.1 vCPU
- Web: Nginx serving static files, minimal

Scale horizontally by adding more worker instances for higher email throughput.
